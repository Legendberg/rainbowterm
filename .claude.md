## CRITICAL: Config Sync Requirement

**After ANY changes to `config.toml`, ALWAYS sync to the installed config:**

```bash
# macOS
cp config.toml ~/Library/Application\ Support/rainbowterm/config.toml

# Linux
cp config.toml ~/.config/rainbowterm/config.toml
```

This is MANDATORY. The `rt` command uses the platform-specific config by default, NOT the project's `config.toml`. Failing to sync causes:
- New patterns appearing broken when they actually work
- Debugging the wrong config file
- Wasted time on phantom issues

**When to sync:** After every commit that touches `config.toml`, after switching branches, before testing.

## CRITICAL: Regression Testing with Sample Files

**Before AND after adding/modifying ANY pattern in `config.toml`, ALWAYS run the sample files:**

```bash
# Test ALL profiles to catch regressions
# Juniper - show commands and config
for f in tests/networking/juniper/common/*.conf; do cat "$f" | rt --profile juniper; done
for f in tests/networking/juniper/ex/**/*.conf; do cat "$f" | rt --profile juniper; done
for f in tests/networking/juniper/qfx/**/*.conf; do cat "$f" | rt --profile juniper; done

# Other vendors
for f in tests/networking/cisco/ios/*.conf; do cat "$f" | rt --profile cisco; done
for f in tests/networking/arista/eos/*.conf; do cat "$f" | rt --profile arista; done
for f in tests/networking/versa/sdwan/*.conf; do cat "$f" | rt --profile versa; done
```

**Why this matters:**
- New patterns can break existing coloring due to priority conflicts
- Regex changes may have unintended side effects
- Visual inspection catches issues automated tests miss

**Sample file requirements:**
- Must cover ALL pattern types for that vendor/model
- Include edge cases and boundary conditions
- Represent real device output (not synthetic)
- Use `.conf` extension for sample files
- Organized by: `tests/networking/<vendor>/<series>/<model>/`

**File naming convention:**
- Show commands: `sample1.conf` or `<hostname>_prechecks_<date>.conf`
- Config hierarchical: `hierarchical.conf` or `<hostname>_configuration_<date>.conf`
- Config set: `set.conf` or `<hostname>_configuration_set_<date>.conf`
- Config inheritance: `inheritance.conf`
- Config diff: `diff.conf`

Descriptive naming (hostname/date) preferred for traceability.

**Collecting samples:**
1. Use `commands.txt` in each vendor directory as the command reference
2. Run PART 1 (show commands) in sequence → save to `sample1.conf`
3. Run PART 2 (config commands) → save to individual files as noted
4. Multiple show samples per model encouraged for coverage

**When adding new patterns:**
1. FIRST: Run existing samples, note current coloring
2. Add pattern to `config.toml`
3. Sync config to platform path
4. Run samples AGAIN - verify new pattern works AND nothing else broke
5. If adding new output types, ADD them to appropriate sample file

This is NON-NEGOTIABLE. Pattern regressions waste hours of debugging.

## Project Overview

**RainbowTerm** is a high-performance Rust CLI tool that provides context-aware syntax highlighting for network device output. It's designed for network engineers who work with routers and switches via SSH.

## Key Concepts

### Terminology: Semantic vs Context-Aware

- **Semantic Coloring** (the philosophy): Color based on *meaning*, not just pattern. The same value can have different colors depending on what it represents (e.g., `1298458` as traffic bytes = blue, as error count = red)
- **Context-Aware** (the mechanism): State machine that tracks multi-line context to enable semantic coloring (e.g., remembers "interface is UP" so subsequent "Half-duplex" = red warning)

### Unique Features
- **Magnitude Spectrum Coloring**: Numbers are colored by magnitude (hundreds→purple, thousands→blue, millions→green, etc.) for instant visual comprehension of packet/byte counters
- **Context-Aware State Machine**: Tracks multi-line state to enable semantic coloring across lines
- **Profile System**: Vendor-specific patterns with inheritance (base → juniper/cisco/arista)
- **Embedded Config**: Works immediately after install, no setup required

### Architecture
```
src/
├── main.rs      - Core engine, pattern matching, CLI
├── config.rs    - TOML parsing, profile management, pattern compilation
├── context.rs   - State machine for context-aware rules
├── convert.rs   - ChromaTerm YAML → TOML converter (optional 'convert' feature)
└── lib.rs       - Library interface

config.toml      - Default config (embedded via include_str!)
tests/           - Vendor-specific test files (no SSH required!)
```

### Adding New Patterns
1. Add regex pattern to `config.toml` under appropriate profile
2. Set priority (200=critical, 100=interfaces, 90=states, etc.)
3. Test with: `cargo run -- --profile <name> < tests/<vendor>-sample.txt`
4. Sync config: `cp config.toml <platform-config-path>/config.toml`
5. Rebuild: `cargo install --path .`

### Configuration Priority System
- **200+**: Critical keywords (error, warning, failure)
- **168**: Error counter zeros (green = healthy)
- **166-160**: Error spectrum (trillions down to hundreds)
- **155-150**: Dropped packets column (error spectrum)
- **155-145**: Neutral spectrum (trillions down to hundreds) and queue counters
- **100**: Interface names
- **90**: Protocol states (BGP, OSPF, STP)
- **85**: Routing markers, roles
- **80**: Speed indicators
- **10**: Generic up/down

## Important Files

### config.toml Structure
```toml
[palette]           # Color definitions (#hex)
[profiles.base]     # Universal patterns
[profiles.juniper]  # Inherits from base
[profiles.cisco]    # Inherits from base
```

### Pattern Format
```toml
[[profiles.vendor.patterns]]
description = "Human-readable description"
regex = '''regex pattern'''
color = "palette-color-name"
priority = 100
[profiles.vendor.patterns.groups]  # Optional: per-capture-group colors
1 = "color-for-group-1"
```

## Known Issues / Gotchas

1. **Config Caching**: `include_str!` embeds config at compile time. After editing `config.toml`:
   - Users with existing config need to update it manually
   - Or delete it to use embedded default
   - Fresh installs always get latest embedded config

2. **Development Config Drift**: ⚠️ **SEE CRITICAL SECTION AT TOP OF FILE** ⚠️
   - The installed config at platform-specific path is used by default
   - Project's `config.toml` changes are NOT automatically installed
   - **ALWAYS sync after config changes**

3. **Cargo Install**: Must use `--path .` for local changes, or `--git` for GitHub

4. **Test Files**: Arista test file exists but profile implementation is incomplete

5. **TOML Syntax**: Watch for nested sections - new `[profiles.vendor]` must be at top level, not inside another profile's arrays

## Project Philosophy

1. **Zero Configuration**: Works immediately after install
2. **Visual Clarity**: Colors convey meaning (magnitude, state, criticality)
3. **Semantic Coloring**: Color based on *meaning*, not just keywords. The same word can mean different things:
   - `up down` (admin up, link down) = 🔴 Red (problem)
   - `down down` (admin down, link down) = 🟡 Yellow (intentional, but notable)
   - `up up` = 🟢 Green (healthy)
   - Achieved via context-aware state machine that tracks meaning across lines
4. **Vendor Aware**: Respect syntax differences between vendors
5. **Developer Friendly**: Test without hardware access
6. **Performance**: No lag on large outputs (SSH compatibility)
