## What this project is

**RainbowTerm** is a Rust CLI (`rt`) that colorizes network device output
(Juniper, Cisco, Arista, Versa) via regex patterns loaded from an embedded
TOML config. Reads stdin, writes colorized stdout. No network, no shell
execution, no writes outside `$CONFIG/rainbowterm/` + rc files on opt-in.

## Core concepts

- **Semantic coloring** (philosophy): color based on meaning, not token. The
  number `1298458` is blue as a traffic counter, red as an error count.
- **Context-aware** (mechanism): state machine tracks multi-line context so
  the same keyword can render differently (`Half-duplex` on an UP link = red,
  on a DOWN link = gray).
- **Magnitude spectrum**: numbers colored by size (hundreds→purple,
  thousands→blue, millions→green, …). Two spectra — neutral for traffic,
  error for drops/errors.
- **Profile inheritance**: `base` (universal patterns) is inherited by every
  vendor profile.

## Architecture

```
src/
├── main.rs      - CLI, stdin loop, output rendering
├── config.rs    - TOML parse, profile resolution, auto-detect scoring
├── matching.rs  - Regex compilation, pattern application
├── context.rs   - State machine for context-aware rules
├── versions.rs  - Config version tracking, blake3 hash validation
└── lib.rs       - Library interface (config, context, matching)

config.toml      - Default config, embedded via include_str! at compile time
tests/networking/ - Realistic vendor output samples (visual regression)
benches/         - Criterion benchmarks
```

## Profiles

| Profile  | Status  | Notes                                         |
|----------|---------|-----------------------------------------------|
| base     | shipped | Universal: IPs, MACs, dates, generic status   |
| juniper  | primary | EX/QFX/MX/SRX — interfaces, BGP, OSPF, STP    |
| cisco    | shipped | IOS/IOS-XE/NX-OS — interfaces, BGP, OSPF, STP |
| versa    | shipped | SD-WAN FlexVNF — interfaces, VRFs, SLA        |
| arista   | partial | Sample files exist, profile config incomplete |

Vendor priority when adding coverage: **Juniper → Versa → Cisco → Arista**.

Default profile is `base` (fallback when auto-detection fails). Silent wrong
coloring is worse than no coloring — this is intentional.

## Pattern priority reference

Higher runs first:

| Priority | Purpose                                        |
|----------|------------------------------------------------|
| 200+     | Critical keywords (error, warning, failure)    |
| 168      | Error counter zeros (green = healthy)          |
| 166–160  | Error spectrum (trillions → hundreds)          |
| 155–150  | Dropped-packets column (error spectrum)        |
| 155–145  | Neutral spectrum + queue counters              |
| 100      | Interface names                                |
| 90       | Protocol states (BGP, OSPF, STP)               |
| 85       | Routing markers, STP roles                     |
| 80       | Speed indicators                               |
| 10       | Generic up/down                                |

## Pattern format

```toml
[[profiles.vendor.patterns]]
description = "Human-readable description"
regex = '''regex pattern'''
color = "palette-color-name"
priority = 100

[profiles.vendor.patterns.groups]  # Optional: per-capture-group colors
1 = "color-for-group-1"
```

## Agent principles

These override defaults when working on this project.

### Verify, don't claim

- If you changed code, run `cargo test` before calling it done.
- If you changed `config.toml`, also sync (`rt --update-config`) and run the
  vendor sample files. Color regressions don't show up in `cargo test`.
- If you can't verify something directly, say so — don't present assumptions
  as verified fact.

### Error messages teach

Every `.with_context()` and `anyhow::bail!` should include: what operation
failed, what path was involved, and what the user should do. Bare `?` on
`fs::*` or `toml::*` is a regression — wrap with the file path.

Pattern:

```rust
std::fs::read_to_string(&path)
    .with_context(|| format!("reading user config {}", path.display()))?
```

Invalid regex in user config is a **hard error at startup**, not a warning.
Warnings get lost in piped stdout. See `Config::validate()`.

### Brian Irvin quality bar

- Security visible at repo root (SECURITY.md exists — keep it current).
- Documentation tables beat prose.
- Pragmatic simplicity — delete dead code rather than gate it behind
  speculative features. The old `convert` module was removed for this reason.
- Read/write separation — write operations are annotated with `// WRITE:`
  doc comments and enumerated in SECURITY.md.

### Comments

Default to writing none. Only add one when the *why* is non-obvious: hidden
constraint, subtle invariant, workaround for a specific bug. Don't explain
the what (code does that), don't narrate the current fix.

### Shell

No heredocs in Bash tool calls. Use `printf '%s\n' "..."` or write to a file
first. Heredocs get stuck in interactive mode on this harness.

### Semantic over syntactic

Always ask "what does this token mean *in this context*?" before coloring.
If the same keyword means different things in different states, that's a
job for `[[profiles.X.contexts]]` state machine rules, not pattern priority.

## Config modification

See [CONTRIBUTING.md § Modifying config.toml](CONTRIBUTING.md#modifying-configtoml)
for the full checklist. The short version:

1. Bump `# Config version: X.Y.Z (YYYY-MM-DD)` header
2. `cargo install --path . && rt --config-hash`
3. Add the printed hash to `src/versions.rs::KNOWN_HASHES`
4. Sync locally: `rt --update-config`
5. Run sample files visually

Never skip #3 — missing hash entries make every user get a prompt instead
of an auto-update.

## Regression testing

Sample files are the source of truth for visual correctness. Run them before
*and* after any pattern change:

```bash
for f in tests/networking/juniper/common/*.conf \
         tests/networking/juniper/ex/*/*.conf \
         tests/networking/juniper/qfx/*/*.conf \
         tests/networking/cisco/ios/*.conf \
         tests/networking/versa/sdwan/*.conf; do
    echo "=== $f ==="
    cat "$f" | rt
done
```

Automated tests (`cargo test`) cover parsing, priority, inheritance, and
byte-level UTF-8 buffer handling. They do **not** verify rendered colors —
that's the sample-file pass.

## Known issues / gotchas

1. **Auto-detect fallback is intentional**: when `rt` can't identify a
   device, it falls back to the `base` profile (universal patterns only) and
   prints a warning to stderr. Per-vendor fallback was tried before; silent
   wrong coloring was worse than no coloring. If you want more aggressive
   detection, add hostname prefixes to `[hostname_prefixes]` in
   `config.toml` rather than changing the fallback behavior.

2. **Config takes precedence over embedded**: the installed user config at
   the platform path overrides the project's `config.toml`. Changes to the
   project config won't appear in the installed `rt` until you run
   `rt --update-config`.

3. **Arista profile is a stub**: sample files exist, profile rules don't.
   Completing it is on the roadmap but not high priority.

4. **TOML nesting**: a new `[profiles.vendor]` must be at the top level, not
   nested inside another profile's arrays. TOML ordering matters.

5. **`cargo install`**: use `--path .` for local changes; `--git` for GitHub.
   `cargo install rainbowterm` fetches from crates.io.

## Code style quick reference

- Rust 2021 edition
- Error handling: `anyhow::Result` with `.with_context()`
- Error display: top-level printer uses `{:#}` to walk the cause chain
- Regex patterns in TOML: raw strings `'''pattern'''`
- Release build: LTO, stripped, optimized for size
- Line length: follow `cargo fmt` defaults

## When the user asks about colors in output

The user can see terminal colors directly — describe what you observe in the
output, don't speculate. If they paste output and ask why something is a
particular color, match it to the responsible pattern in `config.toml` by
priority.

## Testing philosophy

Sample files are comprehensive realistic vendor output. They let you develop
without SSH access to real devices. When you add a new pattern type, add
representative output to the matching sample file — otherwise the next
contributor can't regression-test what you shipped.
